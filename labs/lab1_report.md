# Лабораторна робота №1

## Тема

Збір функціональних і нефункціональних вимог для проєкту **Fitness Club Management System**.

## Мета роботи

Уточнити ролі користувачів, сформувати MVP-обсяг системи та зафіксувати
нефункціональні вимоги, які впливають на архітектуру та подальшу реалізацію.

## Опис предметної області

Fitness Club Management System - це вебзастосунок для роботи фітнес-клубу, який
об'єднує клієнтів, тренерів, адміністраторів і власника в одному цифровому контурі.
Система покриває реєстрацію та логін, перегляд розкладу, бронювання занять, купівлю
абонементів, управління платежами, аналітику та фонові нагадування.

## Зацікавлені сторони

| Сторона | Потреба | Очікуваний результат |
|---|---|---|
| Клієнт | швидко переглядати розклад і купувати абонементи | проста self-service взаємодія |
| Тренер | бачити власні класи і список записаних клієнтів | актуальний робочий графік |
| Адміністратор | керувати користувачами, оплатами та планами | єдина точка операційного контролю |
| Власник | отримувати бізнес-звіти по клубу | прозора аналітика доходів і навантаження |
| Команда розробки | мати підтримувану архітектуру | модульний код і чіткі контракти |

## Функціональні вимоги MVP

| ID | Роль | Вимога | Критерії приймання |
|---|---|---|---|
| FR-01 | Клієнт | Реєстрація та логін через email/password | email валідний, пароль хешується, логін створює cookie-based session |
| FR-02 | Клієнт | Перегляд розкладу | відображається список/календар занять, є фільтрація |
| FR-03 | Клієнт | Покупка абонемента | користувач обирає план, створюється запис платежу, абонемент активується |
| FR-04 | Клієнт | Запис на заняття | не можна перевищити `capacity`, дубльований запис заборонений |
| FR-05 | Клієнт | Перегляд власних бронювань та абонементів | видно поточні й завершені записи, а також статус абонемента |
| FR-06 | Клієнт | Скасування запису | доступне не пізніше ніж за 2 години до старту, візит повертається для лімітованих планів |
| FR-07 | Тренер | Перегляд власного розкладу | тренер бачить лише свої заняття |
| FR-08 | Адміністратор | Керування користувачами | доступні перегляд, створення, редагування і видалення профілів |
| FR-09 | Адміністратор | Керування розкладом | доступний CRUD занять |
| FR-10 | Адміністратор | Фіксація оплат | підтримуються принаймні `CARD` і `CASH` |
| FR-11 | Адміністратор | Заморозка абонемента | можна подовжити `end_date` на період паузи |
| FR-12 | Власник | Звіт по доходах | повертаються агреговані суми за період |
| FR-13 | Власник | Звіт по популярності тренерів | формується рейтинг на базі підтверджених бронювань |
| FR-14 | Система | Нагадування про завершення абонемента | фоновий процес перевіряє підписки і створює reminder flow |

## Нефункціональні вимоги

| ID | Категорія | Вимога | Метрика |
|---|---|---|---|
| NFR-01 | Performance | сторінка розкладу має відкриватися швидко | до 2 секунд після прогріву API |
| NFR-02 | Performance | web та API мають масштабуватись окремо | контейнери ізольовані |
| NFR-03 | Reliability | міграція не повинна ламати старий функціонал | strangler-підхід |
| NFR-04 | Reliability | фонові задачі мають працювати окремо від API | worker + beat |
| NFR-05 | Security | паролі зберігаються лише у вигляді хешів | Argon2id |
| NFR-06 | Security | сесія має бути захищена | Access 15 хв, Refresh 7 днів, HttpOnly cookies |
| NFR-07 | Security | адмінські сесії мають завершуватися по idle timeout | 30 хв неактивності |
| NFR-08 | Security | мутації API мають бути захищені | CORS allowlist, CSRF, rate limit |
| NFR-09 | Usability | увесь UI має бути українською | локалізовані тексти |
| NFR-10 | Compatibility | застосунок має працювати у сучасних браузерах | останні 2 стабільні версії |
| NFR-11 | Compatibility | UI має бути адаптивним | без горизонтального скролу від 320px |
| NFR-12 | Maintainability | код повинен бути модульним | чіткі шари, відсутність fat functions |
| NFR-13 | Observability | потрібні request id і health probes | structured logs, `/health/live`, `/health/ready` |

## Обмеження та припущення

- Перший реліз фокусується на MVP, без social login та password reset.
- Для локальної розробки використовується Docker Compose.
- У системі є чотири ролі: `CLIENT`, `TRAINER`, `ADMIN`, `OWNER`.
- Основний канал взаємодії - вебзастосунок, а не мобільний клієнт.

## Короткі відповіді для захисту

1. Функціональні вимоги описують, **що** система робить, а нефункціональні - **як саме** вона має це робити.
2. Для збору вимог доречні інтерв'ю, спостереження, аналіз бізнес-процесів, прототипування та рев'ю сценаріїв.
3. User Story зазвичай має форму: "Як `<роль>`, я хочу `<дію>`, щоб `<цінність>`".
4. Нефункціональні вимоги критичні, бо саме вони визначають безпеку, швидкодію й підтримуваність корпоративного рішення.
5. Acceptance Criteria перетворюють вимогу на перевірювану умову готовності.

## Висновок

На етапі Лабораторної №1 для проєкту сформовано повний базовий набір ролей,
MVP-вимог і нефункціональних обмежень. Ці вимоги безпосередньо лягли в основу
архітектури, REST API, механізмів безпеки та клієнтського SPA.
